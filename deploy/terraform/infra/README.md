# Terraform Infrastructure

Infrastructure is split into two independently deployable layers:

- **`infra/terraform/`** (this directory) — AWS infrastructure: VPC, EKS, RDS, ACM, Secrets Manager
- **`infra/terraform/application_layer/`** — Kubernetes workloads: ALB controller, external-dns, External Secrets Operator, accounting app (all via Helm)

Both share the same S3 state bucket but use separate state keys.

## Prerequisites

- [Terraform](https://developer.hashicorp.com/terraform/install) >= 1.5
- [AWS CLI](https://docs.aws.amazon.com/cli/latest/userguide/getting-started-install.html) v2
- [kubectl](https://kubernetes.io/docs/tasks/tools/)
- AWS credentials configured (`aws configure` or environment variables)
- DNS zone for `pawel.dev` (managed externally — not in this Terraform)

## 1. Bootstrap Remote State

Creates the S3 bucket and DynamoDB table for Terraform state. Run once.

```bash
cd infra/terraform/bootstrap
terraform init
terraform apply
```

This creates:
- S3 bucket `pawel-kick-accounting-terraform` (versioned, encrypted, public access blocked)
- DynamoDB table `pawel-kick-accounting-terraform-locks`

## 2. Deploy Infrastructure Layer

Provisions VPC, EKS cluster, RDS, ACM certificate, and Secrets Manager entries.

```bash
cd infra/terraform
terraform init
terraform apply \
  -var="google_sa_key_json=$(cat ../../google-sa-key.json)"
```

First apply takes ~15-20 minutes (EKS cluster creation is the bottleneck).

### ACM Certificate Validation

After the first apply, Terraform outputs the CNAME records you need to create in your external DNS provider:

```bash
terraform output acm_validation_records
```

Create the displayed CNAME record(s) in your DNS provider. The certificate will remain in `PENDING_VALIDATION` until the records propagate. The ALB ingress won't serve HTTPS until the cert is validated.

### Infrastructure variables

| Variable | Default | Description |
|---|---|---|
| `aws_region` | `us-west-2` | AWS region |
| `cluster_name` | `accounting` | EKS cluster name |
| `domain_name` | `accounting-8cvbj8765rv.pawel.dev` | App domain |
| `google_sa_key_json` | `""` | Google SA key JSON (sensitive) |
| `google_sa_secret_name` | `accounting/google-sa-key` | Secrets Manager name for Google SA key |

## 3. Deploy Application Layer

Deploys all Kubernetes workloads via Helm. Reads infrastructure outputs from the remote state.

```bash
cd infra/terraform/application_layer
terraform init
terraform apply -var="app_image_tag=<git-sha>"
```

### Application layer variables

| Variable | Default | Description |
|---|---|---|
| `aws_region` | `us-west-2` | AWS region |
| `domain_name` | `accounting-8cvbj8765rv.pawel.dev` | App domain |
| `zone_name` | `pawel.dev` | Domain filter for external-dns |
| `app_image_repository` | `ghcr.io/kickapp/wt-pawelszczepanski` | Container image repo |
| `app_image_tag` | `latest` | Container image tag (git SHA) |
| `google_sa_secret_name` | `accounting/google-sa-key` | Secrets Manager name for Google SA key |

### Deploy a new app version

After CI pushes a new image to GHCR:

```bash
cd infra/terraform/application_layer
terraform apply -var="app_image_tag=<new-git-sha>"
```

No need to touch the infrastructure layer for app deployments.

## Secrets Management

Secrets are stored in AWS Secrets Manager and synced to Kubernetes via External Secrets Operator:

- `accounting/db-password` — auto-generated by Terraform (infrastructure layer, RDS module)
- `accounting/google-sa-key` — populated from `google_sa_key_json` variable (infrastructure layer)

No secrets are passed through Helm values. The ESO `ExternalSecret` resource syncs them into a K8s Secret every hour.

## Connect to the Cluster

```bash
aws eks update-kubeconfig --name accounting --region us-west-2
kubectl get pods -n accounting
```

## Teardown

Destroy application layer first, then infrastructure.

```bash
# 1. Destroy application layer (Helm releases)
cd infra/terraform/application_layer
terraform destroy

# 2. Destroy infrastructure (VPC, EKS, RDS, etc.)
cd infra/terraform
terraform destroy

# 3. Destroy state bucket (optional, only if decommissioning entirely)
cd infra/terraform/bootstrap
terraform destroy
```

```
┌─────────────────────────────────────────────────┐
│ VPC 10.2.0.0/16                                 │
│                                                   │
│  Public Subnets          Private Subnets          │
│  ┌──────────────┐       ┌──────────────────────┐ │
│  │ ALB (ingress)│──────▶│ EKS Node Group       │ │
│  │              │       │  ┌─────────────────┐  │ │
│  └──────────────┘       │  │ accounting-app  │  │ │
│                         │  │ (2 replicas)    │  │ │
│                         │  └────────┬────────┘  │ │
│                         │           │           │ │
│                         │  ┌────────▼────────┐  │ │
│                         │  │ RDS PostgreSQL  │  │ │
│                         │  │ db.t4g.micro    │  │ │
│                         │  └─────────────────┘  │ │
│                         └──────────────────────┘ │
└─────────────────────────────────────────────────┘

DNS: accounting-8cvbj8765rv.pawel.dev → ALB (via external-dns)
TLS: ACM certificate (DNS validation — create CNAME in external DNS provider)
Secrets: AWS Secrets Manager → External Secrets Operator → K8s Secret
```

## State Layout

| State key | Directory | What it manages |
|---|---|---|
| `accounting-app/terraform.tfstate` | `infra/terraform/` | VPC, EKS, RDS, ACM, Secrets Manager |
| `application-layer/terraform.tfstate` | `infra/terraform/application_layer/` | Helm releases (ALB controller, external-dns, ESO, app) |

## Modules

| Module | What it creates |
|---|---|
| `modules/vpc` | VPC, public/private subnets, NAT gateway, internet gateway |
| `modules/eks` | EKS cluster, managed node group, IRSA roles (ALB controller, external-dns, ESO) |
| `modules/rds` | RDS PostgreSQL instance, security group, DB password in Secrets Manager |
| `modules/dns` | ACM certificate (validation records output for external DNS) |
